# BUNKER — AAA Worldbuilding & Production Vision (Canonical)

## 0. Canon Status
This document is the single source of truth for BUNKER’s world, tone, and diegetic governance.
Anything that violates these rules is rejected.

---

## 1. Creative Pillars (Non‑Negotiable)

### 1.1 Diegetic First
Every interface element is an in‑world object: panel, terminal, gauge, placard, console, schematic.
If it cannot exist inside a hardened facility, it does not exist.

### 1.2 Institutional Confidence
The facility never pleads, jokes, or “assists.”
It reports. It authorizes. It executes. It contains.

### 1.3 Mechanical Truth
Systems have inertia. Calibration takes time. State changes are explicit.
Feedback is physicalized: relays, lamps, progress ticks, counters, stamps.

### 1.4 Cinematic Restraint
Atmosphere over spectacle.
Silence over noise.
One strong signal beats ten decorative effects.

### 1.5 Operator Authority
The operator is in command. The system is a toolset, not a companion.

---

## 2. High‑Level Thesis
BUNKER is a local‑first AI command facility designed to protect and empower its operator against the “external dependency frontier”:
unreliable networks, opaque services, cost volatility, and vendor lock‑in.

BUNKER is not software you browse.
BUNKER is infrastructure you operate.

---

## 3. Aesthetic Synthesis (Fallout‑adjacent without infringement)
**Civil‑defense‑era institutional optimism** fused with **post‑crisis industrial decay**:

- Mid‑century signage logic: calm, instructional, authoritative
- Brutalist / industrial material honesty: concrete, steel, bolts, paint wear
- CRT/terminal ergonomics: phosphor glow, scanlines, burn‑in, monospaced telemetry
- Mechanical signaling: status lamps, rotary selectors, lever toggles
- Warning doctrine: hazard stripes, placards, stamped labels, inspection tags

No proprietary names, motifs, mascots, or copyrighted layouts. BUNKER is original infrastructure disguised as fiction.

---

## 4. Tone Rules (Language + Behavior)

### 4.1 Language Lexicon (Required)
- Operator (not user)
- Facility (not app)
- Module (not feature)
- Initialize / Deinitialize (not start/stop)
- Diagnostics (not settings)
- Logs / Telemetry (not activity feed)
- Incident (not error)
- Authorization (not permissions)
- Containment / Lockdown (not “restricted mode”)

### 4.2 Prohibited Tone
- Friendly banter
- Cheerleading
- Apologies as personality
- Modern SaaS microcopy

### 4.3 Feedback Doctrine
Feedback is mechanical and declarative:
- “CALIBRATION COMPLETE”
- “ROUTING TABLE UPDATED”
- “INCIDENT: IO VOLTAGE OUT OF RANGE”

No emojis. No cuteness. No conversational hedging.

---

## 5. World Rules & Governance

### 5.1 The Facility Is Real
BUNKER is a place with:
- Physical rooms
- Power routing
- Air handling
- Storage
- Access control
- Maintenance doctrine
- Operational procedures

If a capability cannot be located in a room, it is not designed yet.

### 5.2 The Facility Is Lived In
**BUNKER is not a sterile hospital. People LIVE here.**

Humanizing elements prevent the facility from feeling dead:
- Potted plants in common areas (morale/survival function)
- Furniture in circulation spaces (worn but maintained)
- Personal artifacts on desks and notice boards
- Operational signage (wet floor signs, laundry labels)
- Warmth from tungsten lighting mixing with fluorescent

The industrial shell contains human life. Without this, it's a tomb.

### 5.3 State Is Explicit
Global state is always legible:
- Stable
- Processing
- Degraded
- Fault
- Offline

State changes are visible (lamps, stripes, stamped logs).

### 5.4 Resource Realism
Compute, storage, time, and uplink usage are treated as consumables.
External relays are optional, metered, and always logged.

### 5.5 Minimal Wordmark Doctrine
“BUNKER” text is rare and ceremonial:
- Boot / initialization
- Facility overview
- Documentation

Everything else is insignia + labeling.

### 5.6 Canon Gate (Rejection Authority)
Any document, asset, UI, or copy that violates:
- Diegetic integrity
- Tone rules
- Mechanical truth
- Operator authority
is rejected.

## 6. Visual Identity (Foundational, not final art)

### 6.1 Primary Mark: Hatch Sigil
A facility insignia (not a “logo”): circular hatch/door geometry, monogram, embedded status lamp.
The sigil also functions as the global system‑state indicator.

### 6.2 Materials & Surfaces
- Concrete: stained, matte, chipped
- Painted steel: scuffed edges, bolt plates, weld seams
- Plastics: aged, slightly yellowed, engraved legends
- Glass: thick, dusty, “instrument” not “smartphone”

### 6.3 Color as Function
Color is never decorative. It encodes state and doctrine.

**Vault Yellow/Gold is the signature accent.** This is THE color that makes BUNKER read as BUNKER:
- Safety railings, door trim, zone numerals, floor markings
- Yellow against steel blue-grey is the foundational contrast pairing
- Without the yellow, it's just a grey industrial space

A restrained bunker palette with CRT accents and warning bands.

### 6.4 Typography Doctrine
- Condensed industrial sans for headings/placards
- Monospace terminal face for logs, metrics, identifiers
- Numbers carry authority (tabular alignment, fixed‑width telemetry)

---

## 7. Systems Realism: How AI Exists Here
AI is infrastructure housed in secured bays:
- Local cores (primary)
- External uplink relays (optional)
- Router tables decide routing based on policy, load, and mission class
- Every run produces a trace: input stamp → route → execution → output → archive

This maps directly to rooms, panels, and bays.

---

## 8. Long‑Term IP Vision (Production Horizon)
BUNKER expands like a living facility:
- New modules are new wings
- New capabilities require new physical constraints (power draw, cooling, access)
- Operator progression is facility upgrade, not “levels”

Future expansions may include:
- Secure communications bay
- Research archive stacks
- Fabrication/repair bay
- Field relay kits
- Simulation chamber (scenario runs)

All remain diegetic.

---

## 9. Rejection Criteria (Immediate No)
Reject any work that reads as:
- Web dashboard/SaaS
- Modern minimalist “clean app”
- Over‑animated, playful, or decorative
- UI without a physical metaphor
- Systems without explicit state, logs, and constraints
